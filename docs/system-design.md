# System Design: Мультиагентная система обучения Windchaser

**Версия**: 1.0  
**Статус**: PoC / Готов к реализации  
**Дата**: апрель 2026

---

## 1. Ключевые архитектурные решения

### 1.1 Мультиагентность с изоляцией контекстов

Система разделена на два полностью независимых агента: **Tutor Agent** и **Evaluator Agent**. Агент-оценщик никогда не получает доступ к истории диалога агента-тьютора в рамках одной сессии — он работает исключительно с финальной работой студента и эталонной рубрикой. Это устраняет конфликт ролей «помощника» и «судьи» и обеспечивает объективность оценивания.

### 1.2 Событийная развязка через Redis Streams

Все агенты взаимодействуют исключительно через асинхронный event bus (FastStream + Redis Streams). Прямых вызовов между агентами нет. Это позволяет независимо масштабировать каждый компонент и перезапускать его без влияния на остальные.

### 1.3 Детерминированный retrieval, LLM — только для генерации

TheoryRetriever реализован как детерминированный keyword-search алгоритм поверх индексированных Markdown-файлов. LLM не участвует в поиске контекста — только в его интерпретации и генерации ответа. Это снижает латентность, стоимость и непредсказуемость retrieval-контура.

### 1.4 SCH (Socratic Constraint Hierarchy) в промпте тьютора

Системный промпт агента-тьютора имеет явную иерархию приоритетов: абсолютные запреты (приоритет 1) → педагогические предписания (2) → адаптивные исключения (3). Запреты не могут быть сняты никакими инструкциями пользователя или LLM.

### 1.5 Двухуровневая оценка в Evaluator Agent

Факт успешности атаки определяется детерминированным **Programmatic Validator** (без LLM). Качественный анализ и обратная связь — **LLMAnalyzer** (Claude 3.5 Sonnet). Разделение гарантирует, что LLM не может «убедить» систему в успехе атаки.

### 1.6 Прямое использование Anthropic SDK без фреймворков

Система использует Anthropic Python SDK напрямую, без LangChain/Langflow. Причина: необходимость явного контроля над иерархией SCH, детальное логирование LLM-вызовов, стабильность API.

## 2. Состав модулей и их роли

| Модуль | Роль |
|---|---|
| **Windchaser Platform** | Образовательная платформа; публикует события студентов в шину, рендерит ответы агентов через WebSocket |
| **AgentBridgeService** | Сервис-переходник внутри Windchaser; транслирует платформенные события в Redis Streams и обратно |
| **FastAPI REST API** | Синхронный управляющий слой: создание/завершение сессий, статус, результаты оценивания |
| **Tutor Agent** | Сократический тьютор; подписан на `student.message`, генерирует направляющие вопросы по SCH |
| **Evaluator Agent** | Агент-оценщик; активируется по `task.submitted`, формирует рубрику и обратную связь |
| **Programmatic Validator** | Детерминированная проверка факта успешности атаки; подписан на `task.attempt` |
| **LLMAnalyzer** | LLM-подмодуль Evaluator Agent; качественный анализ техники по 5-компонентной рубрике |
| **TheoryRetriever** | Keyword-search индекс по Markdown-материалам; предоставляет контекст для системного промпта тьютора |
| **Redis** | Персистентное хранилище диалоговых сессий + брокер сообщений (Redis Streams) |
| **OpenTelemetry + Prometheus** | Сбор трейсов LLM-вызовов, метрик задержек, счётчиков SCR |

## 3. Основной workflow выполнения задачи

### 3.1 Диалог с тьютором (студент работает над заданием)

```
Windchaser → [student.message] → Tutor Agent
    ↓
TheoryRetriever.search(query) → <theory_context>
    ↓
analyze_student_stage(last 5 turns) → {stage, hint_depth}
    ↓
ask_guiding_question(SCH prompt + history) → LLM call
    ↓
Post-processing: вопросительный знак? запрещённые термины? длина?
    ↓
[tutor.response] → Windchaser → WebSocket → студент
```

### 3.2 Попытка атаки (студент отправляет промпт)

```
Windchaser → [task.attempt] → Programmatic Validator
    ↓
Deterministic check: regex / string match / judge model
    ↓
[validation.result] → Windchaser (мгновенная обратная связь)
                    → Tutor Agent (обновление failed_attempts)
```

### 3.3 Финальная сдача задания

```
Windchaser → [task.submitted] → Evaluator Agent
    ↓
Задержка 30 секунд (ожидание записи всех попыток)
    ↓
LLMAnalyzer(final_solution + attempt_log + rubric) → LLM call
    ↓
JSON валидация (Pydantic) + retry up to 2
    ↓
[evaluation.result] → Windchaser → интерфейс курса
```

## 4. State / Memory / Context handling

### Сессионное состояние (TutorSessionState)

Хранится в Redis под ключом `tutor:session:{session_id}:history`.

| Поле | Тип | Описание |
|---|---|---|
| `session_id` | str | Уникальный ID сессии (генерирует Windchaser) |
| `task_id` | str | ID задания |
| `history` | list[{role, content, timestamp}] | Диалог, до 10 последних реплик в LLM-запросе |
| `current_stage` | enum | ORIENTATION / CONCEPT_EXPLORATION / HYPOTHESIS_TESTING / REFINEMENT / SOLVED |
| `hint_depth` | enum | shallow / medium / deep |
| `failed_attempts` | int | Число неуспешных попыток атаки |
| `stage_transitions` | list | История переходов между этапами с метками времени |

**TTL**: 24 часа. При перезапуске сервиса состояние восстанавливается из Redis (персистентность).

### Контекстный бюджет тьютора

- Системный промпт: ~800–1200 токенов (статичные блоки + theory_context)
- История диалога: последние 10 реплик (≈ 2000–3000 токенов)
- Итого входной контекст: < 5000 токенов (с запасом для Claude 3.5 Sonnet 200k)
- Длина ответа: ≤ 512 токенов (принудительное усечение с сохранением финального вопроса)

### Изоляция контекста оценщика

Evaluator Agent не имеет доступа к ключам `tutor:session:*`. Его входные данные: `dialog_log` (из payload события `task.submitted`) + `task:{task_id}:rubric`. Инвариант проверяется автоматическими интеграционными тестами при каждой сборке.

## 5. Описание retrieval-контура (TheoryRetriever)

### Индексация

- Источник: директория с Markdown-файлами учебных материалов (≈14 файлов, ~120k символов)
- Структура данных: forest of section trees (по одному дереву на файл)
- Каждый узел: `heading`, `level`, `content`, `full_content`, `terms`, `breadcrumb`
- Индекс сериализуется в JSON; инкрементальная переиндексация по file mtime < 100 мс

### Поиск

Взвешенная оценка узла:

```
s(v, q) = 1.0 × |T_v ∩ T_q|  +  3.0 × I[heading(v) ⊂ q]  +  0.5 / level(v)
```

- Точное совпадение заголовка (β=3.0) приоритетнее терминального overlap
- Победитель: `argmax s(v, q)` при s > 0; fallback — корневой узел файла темы задания

### Вставка в промпт

Результат вставляется в блок `<theory_context>` системного промпта тьютора при каждом вызове. LLM обязан опираться на этот контекст при формулировке вопросов.

## 6. Tool / API-интеграции

### Внешние API

| Интеграция | Протокол | Описание |
|---|---|---|
| **Anthropic Claude 3.5 Sonnet** | HTTPS REST | Основной LLM; используется для generate (тьютор), classify (analyze_student_stage), analyze (LLMAnalyzer) |
| **Windchaser Platform** | REST + WebSocket | Управляющий REST API (создание сессий, статус, оценки); WebSocket для стриминга ответов в браузер |

### Внутренние сервисные вызовы

| Вызов | Описание |
|---|---|
| `POST /sessions` | Создание новой сессии при начале задания |
| `DELETE /sessions/{id}` | Принудительное завершение |
| `GET /sessions/{id}/status` | Текущий статус + диагностированный этап |
| `GET /evaluations/{id}` | Результат оценивания |
| `POST /theory/reindex` | Ручной триггер переиндексации контента |

### Событийная шина (Redis Streams)

| Канал | Публикует | Потребляет |
|---|---|---|
| `student.message` | Windchaser | Tutor Agent |
| `tutor.response` | Tutor Agent | Windchaser |
| `task.attempt` | Windchaser | Programmatic Validator |
| `validation.result` | Programmatic Validator | Windchaser, Tutor Agent |
| `task.submitted` | Windchaser | Evaluator Agent |
| `evaluation.result` | Evaluator Agent | Windchaser |

## 7. Failure modes, fallback и guardrails

### 7.1 LLM недоступен / превышен rate limit

**Circuit breaker**: при 3 ошибках от Anthropic API за 60 секунд → **аварийный режим**.  
Тьютор возвращает статичный ответ: «Система временно недоступна, попробуйте позже».  
Очередь сообщений не теряется (Redis Streams персистентны).

### 7.2 Нарушение SCH-инварианта (тьютор раскрыл запрещённое)

Post-processing проверяет ответ на наличие терминов из списка абсолютных запретов (конкретные техники атаки). При обнаружении → **ответ отклоняется**, повторный вызов LLM с усиленным ограничивающим инструктажем (до 3 попыток). Если все 3 попытки нарушают инвариант → возвращается neutral fallback («Давайте подумаем вместе — какова цель этого задания?»).

### 7.3 Ответ LLMAnalyzer не прошёл Pydantic-валидацию

Повторный LLM-вызов с явным напоминанием схемы (до 2 retry). При исчерпании попыток → частичный результат с пометкой `evaluation_status: partial`, алерт в мониторинг.

### 7.4 Отсутствие вопроса в ответе тьютора

Post-processing: если в ответе нет `?` → к ответу автоматически добавляется обобщённый уточняющий вопрос + предупреждение в лог. Необходимо для реализации сократического метода в агенте.

### 7.5 Идемпотентность при повторной доставке события

Каждое событие Redis Streams имеет уникальный ID. При повторной доставке (сбой до ACK) → проверяется наличие ранее сохранённого результата в Redis → повторная обработка пропускается.

### 7.6 Jailbreak тьютора студентом

Блок абсолютных запретов (приоритет 1 SCH) не снимается никакими аргументами студента — включая «нехватку времени», «педагогическую нецелесообразность» и прямые просьбы. Попытки jailbreak логируются с меткой `guardrail_triggered`.

### 7.7 Контроль SCR (Socratic Compliance Rate)

Каждый ответ тьютора автоматически классифицируется (наличие `?` → SCR = 1). При SCR < 85% за скользящий час → автоматический алерт. Сигнализирует о деградации промпта или нетипичном паттерне запросов.

## 8. Технические и операционные ограничения

### Latency

| Операция | Целевое значение |
|---|---|
| TheoryRetriever.search | < 50 мс (p95) |
| Tutor Agent: end-to-end (получение → публикация ответа) | < 2 с медиана, ≤ 7 с p95 |
| Programmatic Validator | < 500 мс (p95) |
| LLMAnalyzer (после task.submitted) | ≤ 30 с p95 (включая 30 с задержку) |
| REST API эндпоинты | < 200 мс p95 (без LLM-вызовов) |

### Cost

| Компонент | Оценка |
|---|---|
| Tutor Agent (Claude 3.5 Sonnet, ~5k токенов/вызов) | ~$0.002–0.005 за сообщение студента |
| analyze_student_stage (~800 токенов, T=0.1) | ~$0.0005 за диагностику |
| LLMAnalyzer (финальная оценка, ~3k токенов) | ~$0.003 за сдачу задания |
| **Бюджет пилота (15 студентов)** | $52–95/месяц |
| **Бюджет production (50–100 студентов)** | $230–430/месяц |

Защита от cost overrun: жёсткие лимиты `max_tokens` в каждом LLM-вызове; мониторинг токенной стоимости в логах.

### Reliability

- SLA: ≥ 99% доступности в учебное время
- Fallback при недоступности LLM: статичные ответы (тьютор) / отложенная оценка (оценщик)
- Горизонтальное масштабирование через Kubernetes HPA по метрике «длина очереди Redis Streams»
- Ёмкость минимального production-деплоя: 50 одновременных студентов

### Модели

| Агент | Модель | Температура |
|---|---|---|
| Tutor (генерация вопроса) | Claude 3.5 Sonnet | 0.7 |
| Stage Classifier | Claude 3.5 Sonnet | 0.1 |
| LLMAnalyzer | Claude 3.5 Sonnet | 0.3 |
| Jailbreak Judge (Validator) | Отдельная изолированная модель | 0.0 |
