## Обоснование идеи

Проект — **агентная система для обучения AI/LLM Security**, которая масштабирует менторство и проверку заданий на курсе LLM Security за счёт комбинации RAG и специализированных LLM-агентов.

- **Прикладная задача**:  
  Обеспечить студентам курса доступное и качественное сопровождение (объяснение теории, помощь с практикой, проверка решений), не увеличивая нагрузку на преподавателей.
- **Текущая проблема**:
  - высокая загруженность преподавателей и ограниченное число менторов;
  - высокий спрос на курс, который нельзя масштабировать “ручной” проверкой;
  - ограниченное время на разбор ошибок, что тормозит прогресс в сложной теме безопасности LLM.
- **Гипотеза**:  
  Специализированные агенты (Тьютор и Проверяющий), опирающиеся на RAG и логи автотестов, смогут взять на себя значительную часть консультаций и проверки, сохранив качество и прозрачность.

## Цель и метрики

### Цель проекта

Создать PoC агентной системы (RAG + Агент-Тьютор + Агент-Проверяющий + роутер + простой фронтенд), которая **демонстрирует возможность масштабировать курс LLM Security без пропорционального роста команды** и улучшать качество обучения за счёт персонализированной и объективной обратной связи.

### Продуктовые метрики

- **Пропускная способность курса**:
  - рост числа студентов, которых может сопровождать команда без доп. найма;
- **Успешность прохождения курса**:
  - доля студентов, завершивших курс и сдавших ключевые задания;
- **Удовлетворённость студентов (NPS / рейтинг)**:
  - целевой рост NPS по результатам опроса после курса;
  - средний рейтинг помощи агента ≥ 3/5 на этапе PoC;
- **Активность использования агентов**:
  - среднее количество обращений к агенту-тьютору ≥ 1 на каждое задание.

### Агентские метрики

- **Агент-Тьютор**:
  - удовлетворённость ответами (user rating) ≥ 3/5;
  - доля ответов с корректными ссылками на контекст ≥ 80%;
  - groundedness/faithfulness ≥ 0.8–0.85.
- **Агент-Проверяющий**:
  - совпадение pass/fail с экспертом: accuracy ≥ 0.8, F1 ≥ 0.75;
  - согласованность с экспертом: Cohen’s kappa ≥ 0.7;
  - средний балл по рубрикам фидбэка ≥ 4/5.
- **Роутер интентов**:
  - accuracy ≥ 0.85–0.9 по классам (tutor / evaluator / other);
  - precision/recall по каждому классу ≥ 0.85.

### Технические метрики

- **RAG (retrieval)**:
  - Recall@5 ≥ 0.7, Recall@10 ≥ 0.85;
  - MRR ≥ 0.5–0.6, NDCG@5 ≥ 0.75;
  - groundedness ≥ 0.8 (ответы должны опираться на найденный контекст).
- **Латентность**:
  - p95 retrieval ≤ 600 мс;
  - p95 ответа Тьютора (retrieval + генерация) ≤ 7 с;
  - p95 проверки задания (Проверяющий, включая автотесты) ≤ 30 с;
  - p95 роутера ≤ 100 мс.

## Потенциальные сценарии использования

### Базовые сценарии

- **S1. Вопрос по теории**  
  Студент формулирует вопрос по безопасности LLM (атаки, защитные уровни, threat modeling).  
  - Роутер отправляет запрос к Тьютору;  
  - Тьютор запрашивает контекст через RAG;  
  - даёт ответ с цитатами и задаёт наводящие вопросы.

- **S2. Помощь с практическим заданием**  
  Студент застрял в практическом задании (например, настройка регулярки для L0-защиты или конфигурации ML-классификатора).  
  - Запрос классифицируется как тьюторский;  
  - Тьютор, опираясь на стадию решения студента, описание задания и теорию, помогает шаг за шагом;
  - по возможности избегает “спойлеров”, подталкивая к самостоятельному решению (сократический метод).

- **S3. Автоматическая проверка решения**  
  Студент отправляет решение.  
  - Роутер отправляет запрос к Проверяющему;  
  - Проверяющий при необходимости обращается к теории, анализирует логи и/или структуру решения;  
  - выставляет pass/fail и даёт структурированный фидбэк по рубрикам.

- **S4. Объяснение результата проверки**  
  Студент не понимает, почему задание не прошло.  
  - Повторное обращение (вопрос) уходит Тьютору;  
  - Тьютор опирается на логи/фидбэк Проверяющего и теорию, чтобы объяснить ошибки и подсказать, что улучшить.

### Edge-кейсы

- **E1. Неоднозначный интент запроса**  
  Студент пишет: “Посмотри мой промпт и подскажи, норм ли он”.  
  - Роутер может ошибочно отправить запрос не тому агенту (Tutor vs Evaluator);  
  - fallback: уточняющий вопрос от системы (“вы хотите помощь с доработкой или формальную проверку по критериям?”).

- **E2. Плохой/шумный ввод**  
  Некорректный, слишком короткий или противоречивый вопрос/ответ.  
  - RAG возвращает нерелевантный контекст;  
  - Тьютор обязан явно сигнализировать о недостатке контекста и запросить уточнение, а не галлюцинировать.

- **E3. Атаки на систему**  
  Студент намеренно пытается “сломать” агентов (prompt injection, jailbreak, вытащить секреты и т.п.).  
  - На этапе PoC возможна лишь базовая защита промптами и ограничениями контекста;  
  - edge-кейс: агент не должен раскрывать внутренние системные инструкции и приватные данные, даже если студент провоцирует.

- **E4. Ошибки/нестабильность внешних сервисов (LLM, БД, индексы)**  
  - LLM/хранилища временно недоступны или отвечают с высокой латентностью;  
  - ожидаемая реакция: деградация в сторону статических ответов/ограниченного функционала или прозрачное сообщение об ошибке.

## Ограничения

### Технические (SLO/SLA, latency и т.п.)

- **Доступность**:
  - целевой SLA ≥ 99% в учебное время (для PoC — в минимальном объёме, но с учётом этого ориентира);
  - наличие фолбэка для ключевых операций (например, статические ответы, если RAG не отвечает).
- **Латентность**:
  - p95 retrieval ≤ 600 мс;
  - p95 генерации ответа Тьютора ≤ 7 с;
  - p95 Проверяющего (с автотестами) ≤ 30 с.
- **Безопасность и приватность**:
  - отсутствие PII в логах и тренировочных выборках;
  - изоляция окружений выполнения кода (sandbox);
  - соответствие GDPR/AI Act по прозрачности, traceability и fairness.

### Операционные (бюджет, использование ресурсов)

- **Бюджет LLM и инфраструктуры**:
  - фиксированный бюджет на токены/запросы и compute;
  - приоритизация дешёвых моделей (например, GPT-3.5) там, где не требуется максимальное качество;
  - кеширование retrieval и ответов для частых запросов.
- **Организационные ограничения**:
  - ограниченное время экспертов на разметку данных для Проверяющего и Роутера;
  - необходимость в простоте эксплуатации (docker-compose, стандартные сервисы: PostgreSQL, MinIO и т.п.).

## Архитектурный набросок

Высокоуровневая архитектура PoC включает:

- **Интерфейс**:
  - простой web-фронтенд для студентов (чат с агентами, форма отправки решений, выбор заданий).
- **API-слой / Backend**:
  - API Gateway / FastAPI backend, обрабатывающий запросы от фронтенда;
  - роутер интентов (эвристический/ML), направляющий запросы к нужному агенту.
- **Агенты**:
  - `Агент-Тьютор`: диалог, работа с теорией, RAG, сократический стиль;
  - `Агент-Проверяющий`: анализ решений и логов, работа с рубриками, структурированный фидбэк.
- **RAG и база знаний**:
  - векторный индекс (FAISS) по теоретическим материалам по безопасности LLM;
  - модуль retrieval (например, через LangChain), выдающий top-k фрагментов.
- **Система проверки и рубрик**:
  - модуль автотестов, анализ логов;
  - модуль рубрик (LLM-оценка по критериям).
- **Данные и хранилища**:
  - PostgreSQL (Course DB, LangFlow DB, данные RAG-симулятора);
  - MinIO (логи, отчёты, пользовательские файлы);
  - локальные FAISS индексы (public/private).

## Потенциальный data flow и делегирование LLM/Agent

### Data flow (упрощённый)

1. **Студент → Фронтенд**: вводит вопрос или отправляет решение по заданию.  
2. **Фронтенд → Backend/API**: запрос уходит по HTTP/WebSocket.  
3. **Backend → Роутер**: классификация интента (тьютор / проверка / другое).  
4. **Роутер → Агент**:
   - Тьютор: запрос теории/помощи;
   - Проверяющий: проверка решения.
5. **Агент → Подсистемы**:
   - Тьютор:
     - делает запрос к RAG (векторный поиск по базе знаний);
     - формирует промпт с контекстом и историей диалога;
     - вызывает LLM для генерации ответа и наводящих вопросов.
   - Проверяющий:
     - при необходимости запускает автотесты, читает логи, параметры моделей;
     - формирует структурированный запрос в LLM с рубриками;
     - получает от LLM текстовую обратную связь и/или оценки по критериям.
6. **Ответ → Backend → Фронтенд → Студент**: студент видит ответ, ссылки на источники и рекомендации.

### Что делегируется LLM/Agent

**Делегируется LLM/Agent:**

- семантический поиск и ранжирование контекста (через embeddings + FAISS);
- генерация ответов Тьютора (объяснения, подсказки, наводящие вопросы);
- текстовая интерпретация результатов автотестов и логов;
- формирование структурированного фидбэка по рубрикам;
- (на более поздних этапах) ML-классификация интентов и принятие простых решений pass/fail.

**Не делегируется LLM/Agent (или делегируется частично, под контролем):**

- определение учебных целей и содержимого курса (ответственность преподавателей);
- разработка и утверждение рубрик оценивания и критериев успеха;
- выполнение критичных для безопасности операций (деплой защит, изменение конфигураций прод-систем);
- хранение и обработка PII — остаётся под контролем защищённой инфраструктуры и политик доступа;
- финальное решение по спорным кейсам (human-in-the-loop для high-stakes сценариев).

Таким образом, LLM/агенты берут на себя **контекстуализацию, объяснение и первичную проверку**, а люди-эксперты фокусируются на дизайне курса, сложных кейсах и эскалациях.
