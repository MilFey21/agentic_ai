## Risk register

Таблица рисков для PoC агентной системы обучения AI/LLM Security.

| Риск | Вероятность / Влияние | Детект (как замечаем) | Защита / Митигирующие меры | Остаточный риск |
|------|------------------------|------------------------|-----------------------------|------------------|
| Низкое качество RAG (низкий Recall@k, нерелевантный контекст) | Средняя / Высокое | Мониторинг Recall@k, MRR, NDCG@k на тестовом наборе, ручной просмотр ответов студентов | Использование проверенной embedding-модели, балансировка базы знаний, расширение seed-запросов, гибридный поиск как fallback | Умеренный: возможны отдельные нерелевантные ответы, требуется регулярный мониторинг и дообучение |
| Галлюцинации LLM (неверные ответы Тьютора) | Средняя / Высокое | Жалобы студентов, снижение user rating, автоматические проверки groundedness/faithfulness | Строгие промпты, требование ссылок на источник, ограничение контекста только RAG, human-in-the-loop для high-stakes кейсов | Умеренный: единичные некритичные ошибки допустимы, критичные кейсы должны эскалироваться человеку |
| Низкая точность Агент-Проверяющего (несправедливая оценка) | Средняя / Высокое | Сравнение оценок агента с экспертами (accuracy, F1, Cohen’s kappa), апелляции студентов | Бейзлайн на правилах и автотестах, использование рубрик, проверка на отложенной выборке, ручная проверка спорных кейсов | Низко–умеренный: агент используется как помощник, а не единственный источник истины в пилоте |
| Ошибки классификации интентов (неправильный роутинг) | Средняя / Среднее | Анализ логов маршрутизации, confusion matrix по классам, пользовательские жалобы | Эвристический роутер с прозрачными правилами, уточняющие вопросы при неоднозначных запросах, логирование всех решений роутера | Низкий: некорректный роутинг можно компенсировать уточняющим диалогом |
| Высокая латентность (плохой UX, отваливаются ответы) | Низкая / Среднее | Мониторинг p95 латентности (retrieval, генерация, проверка), алерты по SLA | Кеширование частых запросов и результатов RAG, использование более быстрых моделей, оптимизация контекста, тайм-ауты и деградация функционала | Низко–умеренный: при пиках нагрузки возможны задержки, но без потери критичных функций |
| Высокая стоимость использования LLM (выход за бюджет) | Средняя / Среднее | Мониторинг токенов/стоимости по API, алерты по бюджету | Лимиты на токены/запросы, tiered-модели (дешевле для простых запросов), кеширование, отключение неключевых функций при превышении лимита | Низко–умеренный: при сильном росте нагрузки потребуется масштабирование бюджета или оптимизация сценариев |
| Утечка или некорректная обработка персональных данных | Низкая / Высокое | Аудит логов, периодические проверки соблюдения политик, security review | Строгий запрет PII в логах и тренировочных выборках, псевдонимизация идентификаторов, ограничение доступа, шифрование, политика хранения и удаления | Низкий: PoC минимизирует объём PII и использует обезличенные данные там, где возможно |
| Prompt injection / jailbreak на агентов | Средняя / Высокое | Мониторинг аномальных запросов, ручной разбор подозрительных логов, тесты на атакующие промпты | Жёсткие системные промпты, строгая фильтрация контекста, запрет следовать пользовательским инструкциям, противоречащим политикам, тестовый набор атак для регрессионного тестирования | Умеренный: агенты могут быть частично уязвимы, требуется постоянное улучшение защит и обучение на ошибках |
| Ошибки или падение внешних сервисов (LLM, БД, индексы) | Средняя / Среднее | Алерты по отказам, тайм-аутам, кодам ошибок; мониторинг доступности | Тайм-ауты и ретраи, фолбэк к статическим ответам/ограниченному функционалу, изоляция критичных компонентов, резервные конфигурации | Умеренный: при длительном аутейдже возможно временное снижение функционала или недоступность части возможностей |
| Устаревание базы знаний (неактуальное содержание) | Низкая / Среднее | Жалобы студентов на неточные материалы, периодический контент-ревью, метрики качества RAG | Регулярная пересборка индексов, обновление источников, ревью теоретических материалов экспертами | Низкий: устаревшие куски могут встречаться, но критичные темы обновляются вручную |

## Политика логов и работы с персональными данными

- **Принципы**:
  - минимизация собираемых данных — логируются только необходимые технические события и обезличенные идентификаторы;
  - отсутствие “сырых” персональных данных студентов в логах и тренировочных датасетах;
  - прозрачность и трассируемость действий агентов при сохранении приватности пользователей.

- **Логирование**:
  - события запросов/ответов агентов (timestamp, тип запроса, тип агента, метрики латентности, размер контекста, используемая модель);
  - технические ошибки и исключения (стек-трейсы, коды ошибок, тайм-ауты);
  - решения роутера (к какому агенту направлен запрос и по какой причине — правило/класс);
  - по возможности **не сохраняется** полный текст пользовательского ввода и ответов, либо они агрегируются/анонимизируются для анализа качества;
  - для задач оценки качества возможны отдельные, строго ограниченные выборки диалогов, доступные только команде курса.

- **Персональные данные (PII)**:
  - идентификаторы студентов в логах заменяются на псевдонимы/ID (без ФИО, email и т.п.);
  - доступ к данным о прохождении курса хранится в защищённой БД (PostgreSQL) с аутентификацией и авторизацией;
  - все подключения к БД и MinIO — по защищённым каналам; чувствительные данные шифруются “в покое” и при передаче;
  - соблюдаются требования GDPR/AI Act: право на удаление данных, ограничение сроков хранения, прозрачность использования.

- **Политика хранения**:
  - технические логи хранятся ограниченный период (например, 30–90 дней) для отладки и мониторинга;
  - обезличенные агрегированные метрики могут храниться дольше для аналитики и улучшения систем;
  - проводится регулярная очистка старых логов и архивов в соответствии с политикой хранения.

## Защиты от injection, подтверждение действий и безопасное поведение агентов

- **Защиты от prompt injection / jailbreak**:
  - строгие системные промпты для Тьютора и Проверяющего, явно запрещающие:
    - изменять свои инструкции по требованию пользователя;
    - раскрывать внутренние промпты, ключи, детали конфигурации и приватные данные;
    - выполнять действия вне заданного домена курса;
  - фильтрация контекста: в промпт попадают только проверенные фрагменты из базы знаний и логов, исключая системные и конфиденциальные данные;
  - тестовый набор атакующих промптов (prompt injection, jailbreak, prompt leaking) для регрессионного тестирования и оценки устойчивости;
  - ограничение прав агентов: они не выполняют произвольный код и не имеют прямого доступа к внешним системам вне предусмотренных tools.

- **Подтверждение действий и безопасный UX**:
  - для потенциально критичных действий (например, изменение конфигурации задания, сброс прогресса, массовые операции) требуется явное подтверждение пользователя;
  - при автоматической переоценке или изменении оценки задания система должна:
    - явно уведомлять студента о причинах изменений;
    - в спорных случаях предлагать опцию эскалации к преподавателю;
  - агенты обязаны явно сигнализировать об отсутствии достаточного контекста, а не “выдумывать” ответы.

- **Обработка вредоносного или неуместного контента**:
  - базовая фильтрация ввода (рекламный спам, оскорбления, нерелевантные темы) и указание пользователю на нарушение правил;
  - агенты не дают подробных инструкций по созданию атак за пределами учебного контекста и всегда подчёркивают этическую и легальную рамку курса;
  - логи потенциально вредоносных запросов могут помечаться для отдельного анализа командой безопасности.

В совокупности эти меры должны снизить риски для студентов, преподавателей и инфраструктуры, сохранив при этом ценность агентной системы как инструмента обучения и оценки.
